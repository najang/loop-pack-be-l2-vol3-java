## 1.1 인덱스란?

데이터베이스에서 인덱스(Index)란 **특정 컬럼의 값을 정렬된 순서로 별도 저장해 두는 자료구조**입니다.

책의 색인(목차)을 떠올려 보면 이해하기 쉽습니다. 두꺼운 책에서 특정 용어를 찾을 때, 처음 페이지부터 끝까지 훑는 대신 색인 페이지에서 해당 용어를 찾고 바로 해당 페이지로 이동하죠. 데이터베이스 인덱스도 동일한 원리입니다. **전체 테이블을 풀 스캔(Full Scan)하지 않고, 인덱스를 통해 원하는 데이터의 위치로 빠르게 이동**할 수 있습니다.

### 인덱스의 내부 구조

인덱스는 실제 데이터를 복제해서 저장하지 않습니다. 대신, **인덱스의 끝에는 실제 데이터가 저장된 디스크 주소(포인터)만을 보관**합니다.

```
[인덱스 구조 예시 - 이름 컬럼 기준]

인덱스 (정렬된 키 + 포인터)     실제 데이터 (힙)
------------------------------   -------------------
"김민수" → 포인터(row #3)   →   row #3: {id=3, name="김민수", age=29}
"박지영" → 포인터(row #1)   →   row #1: {id=1, name="박지영", age=34}
"이하늘" → 포인터(row #5)   →   row #5: {id=5, name="이하늘", age=25}
```

이처럼 인덱스는 **키 값과 실제 데이터의 위치 정보만 가지고 있어** 원본 테이블보다 훨씬 가볍고, 정렬된 상태를 유지하기 때문에 이진 탐색(Binary Search) 등을 통해 빠르게 조회할 수 있습니다.

### 조합(Composite) 인덱스와 인덱스 수의 한계

중요한 점은 **정렬 기준이 달라지면 새로운 인덱스가 필요**하다는 것입니다.

예를 들어, `이름` 기준으로 정렬된 인덱스와 `나이` 기준으로 정렬된 인덱스는 완전히 별개입니다. `(이름, 나이)` 복합 인덱스와 `(나이, 이름)` 복합 인덱스도 마찬가지로 다른 인덱스입니다. 자주 사용하는 조회 패턴마다 별도의 인덱스를 만들어야 원하는 성능을 얻을 수 있습니다.

그러나 인덱스를 무분별하게 많이 만들면 **오히려 성능이 저하**될 수 있습니다.

> **인덱스가 많을수록 쓰기 비용이 증가합니다.**
>
> `INSERT`, `UPDATE`, `DELETE` 등 데이터 변경이 발생할 때마다, 변경된 내용을 해당 테이블과 연관된 **모든 인덱스에 반영**해야 합니다. 인덱스가 10개라면 데이터 1건 변경 시 10개의 인덱스를 함께 갱신해야 하는 부담이 생깁니다.

따라서 인덱스 설계는 **조회 성능과 쓰기 비용 사이의 균형**을 고려하는 것이 핵심입니다.

## 1.2 인덱스 설계 원칙

인덱스는 무조건 추가한다고 좋은 것이 아닙니다. **잘못 설계된 인덱스는 추가하지 않은 것만 못할 수 있습니다.** 다음 세 가지 원칙을 기반으로 실제 쿼리 패턴에 맞는 인덱스를 설계해야 합니다.

### 원칙 1: 실제 SELECT 쿼리 패턴에 맞춰 설계하세요

"이 컬럼은 자주 쓰이니까 걸어두면 좋겠지"라는 직관적인 판단만으로 인덱스를 추가하면, 실제로는 전혀 활용되지 않는 인덱스만 쌓이게 됩니다.

인덱스를 설계하기 전에는 반드시 **애플리케이션에서 실제로 실행되는 쿼리를 먼저 분석**해야 합니다. `WHERE` 절에 어떤 컬럼이 오는지, `JOIN` 조건은 무엇인지, `ORDER BY`는 어떤 컬럼을 기준으로 하는지를 파악한 뒤, 그 패턴에 맞게 인덱스를 구성해야 합니다.

```sql
-- 실제로 나가는 쿼리를 먼저 파악하세요
SELECT *
FROM orders
WHERE user_id = 1
  AND status = 'PAID'
ORDER BY created_at DESC;

-- 이 쿼리에 맞는 인덱스: (user_id, status, created_at)
CREATE INDEX idx_orders_user_status_created
    ON orders (user_id, status, created_at);
```

### 원칙 2: 복합 인덱스는 컬럼 순서가 중요합니다 (Leftmost Prefix)

복합 인덱스에서는 **선두 컬럼부터 순서대로 일치해야만 인덱스가 사용**됩니다. 이를 **Leftmost Prefix(최좌선 접두사)** 규칙이라고 합니다.

예를 들어, `(col1, col2)` 복합 인덱스가 있다면:

- `WHERE col1 = ?` → 인덱스 **사용 가능** (col1이 선두 컬럼)
- `WHERE col1 = ? AND col2 = ?` → 인덱스 **사용 가능** (선두부터 순서대로 일치)
- `WHERE col2 = ?` → 인덱스 **사용 불가** (선두 컬럼 col1이 빠졌음)

```sql
-- (user_id, status) 복합 인덱스
CREATE INDEX idx_orders_user_status ON orders (user_id, status);

-- 인덱스 활용 가능
SELECT * FROM orders WHERE user_id = 1;
SELECT * FROM orders WHERE user_id = 1 AND status = 'PAID';

-- 인덱스 활용 불가 (선두 컬럼 누락)
SELECT * FROM orders WHERE status = 'PAID';
```

`(col2, col1)`은 `(col1, col2)`와 **완전히 다른 인덱스**입니다. 따라서 쿼리 패턴에서 가장 자주 등장하고 선택성이 높은 컬럼을 선두에 배치하는 것이 중요합니다.

### 원칙 3: DB 함수가 적용된 컬럼에는 인덱스가 작동하지 않습니다

컬럼에 **함수를 감싸면 인덱스를 타지 못합니다.** 인덱스는 원본 컬럼 값 기준으로 정렬되어 있는데, 함수 적용 후의 값은 인덱스 탐색 시점에 알 수 없기 때문입니다.

```sql
-- 인덱스 사용 불가: created_at 컬럼에 DATE() 함수 적용
SELECT * FROM orders
WHERE DATE(created_at) = '2025-07-04';
```

위 쿼리는 `created_at` 컬럼에 인덱스가 있더라도 **풀 스캔**이 발생합니다. `DATE()` 함수로 감싸는 순간 인덱스 접근이 불가능해지기 때문입니다.

**해결책: 범위 조건으로 대체하세요.**

```sql
-- 인덱스 사용 가능: 범위 조건으로 대체
SELECT * FROM orders
WHERE created_at >= '2025-07-04 00:00:00'
  AND created_at < '2025-07-05 00:00:00';
```

범위 조건으로 작성하면 `created_at` 인덱스를 정상적으로 탐색할 수 있습니다. 날짜 조건 외에도 `YEAR()`, `MONTH()`, `LOWER()`, `UPPER()` 등 함수를 컬럼에 직접 적용하는 경우 모두 동일하게 적용되는 원칙입니다.

## 1.3 인덱스 최적화

> **TL;DR**
>
> `(필터 컬럼, 정렬 컬럼 DESC)` 복합 인덱스를 선택한 이유:
> - 단일 컬럼 인덱스로는 `ORDER BY filesort`를 제거할 수 없기 때문
> - 필터 컬럼을 선두에 두어야 인덱스 탐색 범위가 먼저 좁혀지기 때문
> - 정렬 방향(`DESC`)을 인덱스와 일치시켜야 역방향 스캔 비용을 피할 수 있기 때문

앞서 설명한 인덱스 설계 원칙을 실제 프로젝트에 적용하면 어떤 변화가 생길까요? 이번 섹션에서는 실제 서비스의 두 가지 조회 쿼리에 인덱스를 적용하기 전후를 비교하며, 인덱스 최적화의 효과를 구체적인 수치로 살펴보겠습니다.

### 케이스 1: 브랜드별 상품 목록 조회 (최신 순 정렬)

#### AS-IS: 인덱스 없이 조회

브랜드별 상품 목록을 최신 순으로 정렬해서 반환하는 쿼리입니다.

```sql
SELECT *
FROM products
WHERE brand_id = ?
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 20;
```

이 쿼리에 `EXPLAIN`을 실행하면 다음과 같은 결과가 나옵니다.

| id | select_type | table    | type | possible_keys | key  | rows  | Extra                          |
|----|-------------|----------|------|---------------|------|-------|-------------------------------|
| 1  | SIMPLE      | products | ALL  | NULL          | NULL | 99465 | Using where; Using filesort   |

- `type=ALL`: 전체 테이블을 풀 스캔하고 있습니다. `brand_id`에 인덱스가 없어 모든 행을 하나씩 확인합니다.
- `rows=99465`: 약 10만 건을 전부 읽어야 합니다.
- `Using filesort`: `created_at` 기준 정렬을 위한 별도의 소트 작업이 발생합니다. 인덱스를 통한 정렬이 아니므로 추가 비용이 들어갑니다.

**실행 시간: 약 70ms**

단 20건을 가져오기 위해 10만 건을 전부 읽고, 게다가 파일 정렬까지 수행하는 비효율적인 구조입니다.

#### TO-BE: 복합 인덱스 적용

`brand_id`로 필터링하고 `created_at DESC`로 정렬하는 패턴에 맞춰 복합 인덱스를 추가합니다.

```java
@Table(
    name = "products",
    indexes = {
        @Index(name = "idx_brand_created_at", columnList = "brand_id, created_at DESC")
    }
)
public class ProductModel extends BaseEntity {
    // ...
}
```

인덱스 적용 후 동일한 쿼리에 `EXPLAIN`을 실행하면 결과가 달라집니다.

| id | select_type | table    | type | possible_keys        | key                  | rows | Extra       |
|----|-------------|----------|------|----------------------|----------------------|------|-------------|
| 1  | SIMPLE      | products | ref  | idx_brand_created_at | idx_brand_created_at | 1067 | Using where |

- `type=ref`: 인덱스를 활용해 `brand_id`가 일치하는 행들만 탐색합니다.
- `key=idx_brand_created_at`: 설계한 복합 인덱스가 실제로 사용되고 있습니다.
- `rows=1067`: 탐색 대상이 99,465건에서 1,067건으로 크게 줄었습니다.
- `Using filesort` 제거: 인덱스 자체가 `created_at DESC` 순서로 정렬되어 있으므로 별도의 소트 과정이 필요 없습니다.

**실행 시간: 약 0.805ms**

| 지표          | AS-IS       | TO-BE      | 개선율           |
|--------------|-------------|------------|-----------------|
| 탐색 rows    | 99,465건    | 1,067건    | 98.93% 감소     |
| 실행 시간    | 약 70ms     | 약 0.805ms | 98.85% 감소 (약 87배 개선) |

---

### 케이스 2: 사용자별 좋아요 목록 조회

#### AS-IS: 인덱스 없이 조회

특정 사용자가 좋아요를 누른 상품 목록을 최신 순으로 가져오는 쿼리입니다.

```sql
SELECT *
FROM likes
WHERE user_id = ?
ORDER BY created_at DESC;
```

`EXPLAIN` 결과는 다음과 같습니다.

| id | select_type | table | type | possible_keys | key  | rows  | Extra                          |
|----|-------------|-------|------|---------------|------|-------|-------------------------------|
| 1  | SIMPLE      | likes | ALL  | NULL          | NULL | 99341 | Using where; Using filesort   |

- `type=ALL`: `user_id`에 인덱스가 없어 전체 테이블 풀 스캔이 발생합니다.
- `rows=99341`: 약 10만 건을 전부 검사합니다.
- `Using filesort`: `created_at` 정렬을 위한 별도의 소트 작업이 수행됩니다.

**실행 시간: 약 44.3ms**

#### TO-BE: 복합 인덱스 적용

`user_id`로 필터링하고 `created_at DESC`로 정렬하는 패턴에 맞는 복합 인덱스를 추가합니다.

```java
@Table(
    name = "likes",
    indexes = {
        @Index(name = "idx_user_created_at", columnList = "user_id, created_at DESC")
    }
)
public class LikeModel extends BaseEntity {
    // ...
}
```

적용 후 `EXPLAIN` 결과입니다.

| id | select_type | table | type | possible_keys       | key                 | rows | Extra |
|----|-------------|-------|------|---------------------|---------------------|------|-------|
| 1  | SIMPLE      | likes | ref  | idx_user_created_at | idx_user_created_at | 1000 | -     |

- `type=ref`: `user_id` 인덱스를 통해 해당 사용자의 좋아요 행만 탐색합니다.
- `key=idx_user_created_at`: 설계한 복합 인덱스가 사용되고 있습니다.
- `rows=1000`: 탐색 대상이 99,341건에서 1,000건으로 줄었습니다.
- `Extra` 항목에 `Using filesort`가 사라졌습니다. 인덱스 순서대로 결과를 반환하므로 별도의 정렬 작업이 필요 없습니다.

**실행 시간: 약 0.666ms**

| 지표          | AS-IS       | TO-BE      | 개선율           |
|--------------|-------------|------------|-----------------|
| 탐색 rows    | 99,341건    | 1,000건    | 98.99% 감소     |
| 실행 시간    | 약 44.3ms   | 약 0.666ms | 98.50% 감소 (약 66.5배 개선) |

---

### 정리

두 케이스 모두 **`(필터 컬럼, 정렬 컬럼 DESC)` 형태의 복합 인덱스** 하나만 추가했을 뿐인데, 탐색 행 수가 약 99% 감소하고 실행 시간이 수십 ms에서 1ms 미만으로 줄었습니다.

이처럼 인덱스 최적화의 핵심은 **실제로 나가는 쿼리의 `WHERE` + `ORDER BY` 패턴을 파악하고, 그에 딱 맞는 복합 인덱스를 설계하는 것**입니다. `EXPLAIN`으로 실행 계획을 확인하고 `Using filesort`, `type=ALL` 등의 신호를 읽는 습관을 들이면, 불필요한 전체 스캔과 정렬 비용을 효과적으로 제거할 수 있습니다.
